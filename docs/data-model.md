# KidPlan Data Model

The backend store is a single Google Sheet with 5 tabs (the `Days` tab was removed in Wave 3.5). The GAS API is the only writer; the frontend never talks to Sheets directly. Reads open the Sheet through a single `openSheet_()` chokepoint; writes go through the `upsertRow_`/`deleteRow_` helpers guarded by `LockService.getScriptLock()` (released in a `finally`).

Conventions used throughout:

- Dates are ISO 8601 calendar dates (`YYYY-MM-DD`), stored as text (number format `@`) so the round-trip is timezone-stable.
- Datetimes are ISO 8601 with timezone (`YYYY-MM-DDTHH:mm:ssXXX`).
- Times-of-day are 24-hour `HH:mm`.
- Booleans are stored as the strings `TRUE` / `FALSE` (Sheets native) and normalized to JS booleans on read for columns listed in `BOOLEAN_COLUMNS_`.
- IDs are short opaque strings generated by GAS (`Utilities.getUuid()` truncated).
- "Nullable: yes" means the cell may be empty.

## Tab: `PlanItems`

One row per planned activity. Every activity — primary or backup — lives here. Each row mirrors to a Google Calendar event on the family calendar; backups also live on the family calendar but with a `[Backup]` title prefix and the muted Graphite event color (Google Calendar `eventColor = 8`).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | string | no | Opaque GAS-generated ID. Primary key. |
| `date` | date | no | ISO 8601 (`YYYY-MM-DD`). Stored as text. |
| `kid` | enum (string) | no | One of `shared`, `elder`, `younger`. Defaults to `shared`. Drives lane label on the row + the `[Shared]/[Elder]/[Younger]` prefix on the calendar title. |
| `title` | string | no | Activity title shown in the app and on the calendar event. |
| `description` | string | yes | Free-text. Pre-fills from `Library.description` on add; then independent. Copied to the calendar event description. |
| `start_time` | time | no | 24-hour `HH:mm`. Required by the server — the add flow always collects one. |
| `end_time` | time | yes | 24-hour `HH:mm`. If blank, the calendar layer fills with `start_time + Library.typical_duration_min` if known, else +60 min. Never all-day from the quick path. |
| `location` | string | yes | Free-text (also passed to the calendar event's `location`). |
| `tag` | enum (string) | yes | Optional. Must exist in `Tags.tag` if set. Drives the row's color swatch. |
| `is_backup` | bool | no | `TRUE` if this row is a paired backup; `FALSE` for primaries. |
| `backup_for_id` | string | conditional | When `is_backup = TRUE`, the `PlanItems.id` of the primary this backs up. Empty otherwise. Backups cannot chain (the server rejects a backup whose target is itself a backup). |
| `gcal_event_id` | string | yes | Google Calendar event ID for this row's mirrored event. Populated after the first calendar sync; used for idempotent updates. |
| `source` | enum (string) | no | One of `manual`, `library`, `ocr`, `duplicate`. Tracks how the row entered the system. |
| `updated_at` | datetime | no | ISO 8601 with timezone. Stamped by `upsertRow_` on every write. |
| `item_type` | enum (string) | yes | `''` (normal activity), `nap_one` / `nap_two` (a nap inserted on a one-nap / two-nap day - Google Calendar colors BLUE / PALE_BLUE), or `eldest` (the elder kid's plan paired under a nap). Added by `migrateAddItemTypeColumn` (nap feature, v0.6). |

Pairing semantics:

- Each primary may have at most one paired row: a backup (`is_backup = TRUE`, `backup_for_id = primary.id`, `item_type = ''`) under a normal activity, or an **eldest plan** (`is_backup = TRUE`, `item_type = 'eldest'`, `kid = elder`) under a nap. The server rejects mismatches (a plain backup under a nap, or an eldest plan under a non-nap).
- Eldest plans reuse all backup machinery (cascade, time-drag) but render as a real plan: `[Elder]` calendar title (no `[Backup]` prefix), default event color (no gray).
- Deleting a primary cascades to its paired row + calendar event.
- Deleting a paired row is isolated; the primary is unaffected.
- Editing a primary's `date`, `start_time`, or `end_time` automatically drags the paired row's time + calendar event to match (`upsert_plan_item` handles this server-side).
- Naps are primaries with `kid = younger`, `title = 'Nap'`, no tag. The app inserts them via the "+ Add nap" sheet: two-nap days at 9:00a + 2:00p (90 min each), one-nap days at 11:00a (3 hrs).

## Tab: `Library`

Reusable activities. The picker pulls from here for both primary adds and backup adds.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | string | no | Opaque GAS-generated ID. Primary key. |
| `name` | string | no | Short label (e.g. "Children's museum"). |
| `tag` | enum (string) | no | Must exist in `Tags.tag`. Drives the activity color swatch wherever the item appears. |
| `description` | string | yes | Free-text. Pre-fills into a PlanItem's `description` on add. |
| `indoor` | bool | no | `TRUE`/`FALSE`. Surfaces as an icon in the picker / library card. |
| `typical_duration_min` | number | yes | Integer minutes. Used to fill a PlanItem's `end_time` when the user leaves it blank. |
| `kid_age_fit` | enum (string) | no | One of `any`, `young`, `older`, `both`. Shown in the picker meta line. |
| `notes` | string | yes | Anything else (cost, hours, parking quirks). |

## Tab: `Tags`

Preset + user-added tags. Single source of truth for valid tag values on `Library` and `PlanItems`. Tags are NOT applied to days as a whole — that concept was removed in Wave 3.5.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `tag` | string | no | Primary key. Short lowercase label (e.g. `indoor`, `day-trip`, `mountains`). |
| `color` | string | no | Hex color (`#RRGGBB`) used for the activity swatch and library card. |
| `display_order` | number | no | Integer. Lower numbers shown first in pickers / filter chips. |
| `is_preset` | bool | no | `TRUE` for the shipped seed set; `FALSE` for parent-added tags. |

## Tab: `Photos`

Archive of paper-calendar photos and their OCR reconciliation state (Wave 4).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | string | no | Opaque GAS-generated ID. Primary key. |
| `drive_file_id` | string | no | Drive file ID for the uploaded JPG. |
| `uploaded_at` | datetime | no | ISO 8601 with timezone. |
| `covers_date_range_start` | date | yes | ISO 8601 (`YYYY-MM-DD`), text-stored. |
| `covers_date_range_end` | date | yes | ISO 8601 (`YYYY-MM-DD`), text-stored, inclusive. |
| `ocr_text` | string | yes | Raw Cloud Vision `DOCUMENT_TEXT_DETECTION` output. |
| `parsed_json` | string | yes | JSON string of structured candidate events (date / title / start_time / end_time / kid_hint / location), pre-reconciliation. |
| `reconciled` | bool | no | `TRUE` once every candidate has been accepted, edited, or explicitly rejected. |

OCR-imported PlanItems carry `source = 'ocr'`, `is_backup = FALSE`. The OCR pipeline never produces backups in v1; the user can add a paired backup manually afterward.

## Tab: `Settings`

Singleton key-value store. Mirrors the Script Properties so the frontend can read them via the API without needing Script Property access.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `key` | string | no | Primary key. App-editable keys: `family_calendar_id`, `read_only_calendar_ids`, `photo_drive_folder_id`, `nap_mode` (`one` or `two`; pre-selects the Add-nap sheet's answer, default `two` when unset). Migration also writes `backups_migrated = 'TRUE'` so `migrateToBackupsModelV2` knows it has already run. Secrets (`API_TOKEN`, `VISION_SERVICE_ACCOUNT_JSON`) live ONLY in Script Properties, never in this tab, and are never returned by `get_settings`. |
| `value` | string | no | Stringified value. For `read_only_calendar_ids` this is a comma-separated string. |

## Removed tab: `Days`

The `Days` tab was deleted in Wave 3.5 along with the day-type tag concept, `plan_a_summary`, `plan_b_pointer`, `plan_b_custom_text`, `notes`, `confidence`, and `weather_sensitive`. The product no longer has day-level state — every label and description hangs off an activity. The `migrateToBackupsModelV2` editor function performed the one-time conversion: any legacy Days row with a Plan B pointer or custom text was synthesized into a paired backup on that day's first primary (when an anchor existed), then the tab was deleted.
